## 4. Follow-up (not this change)

### Decided — server requirements (BrandOS MCP; **not implementable in this repo**, handed off)

- [ ] 4.1 **Requirement A — REJECT, never infer.** For `ad` content, `save_post_content` MUST **reject** a call that omits `brief_id` (an `invalid_input`-class refusal) and write nothing; it MUST NOT infer a brief. `post` content is unchanged (the idea's single brief still resolves server-side when omitted; an explicit value still wins on any channel). Rationale: inference is defensible for `post` (one brief, unambiguous) and indefensible for `ad` (N approved angles — the guess is a pick out of several, indistinguishable from a chosen stamp and undetectable downstream). The caller always knows its angle; a server guess can only ever be wrong. Nothing in `plugins/` can satisfy this — the plugin fix (task 1.1) does not wait on it, since an explicit `brief_id` already wins today.
- [ ] 4.2 **Requirement B — PURGE the lineage-less rows.** Existing `content` rows whose `brief_id` **IS NULL** MUST be **deleted**. Such a row is attributable to no angle, can never be safely consumed (it is exactly what makes `/ssc.image` STOP rather than risk grounding a visual in the wrong angle's story), and cannot be repaired by the plugin — which has no record of which angle each historical row was written from. Unusable, not merely untidy. **Run it after A and after `ads-angle-set-curation`'s delete cascade land** — with the write path closed (A) and `content.brief_id ON DELETE SET NULL` replaced by a hard-delete cascade, a NULL `brief_id` on ad content is an impossible state, which makes this a **one-time** cleanup rather than a recurring chore.
