## User Input

```text
$ARGUMENTS
```

Consider the user input above before proceeding (if not empty). Expected inputs — **both required**:

- **Ad concept idea ID** (`idea_id`) — the id of ONE approved ad concept (an `ideas` row, `channel='ad'`, `status='approved'`).
- **Angle brief ID** (`brief_id`) — the id of the operator's **chosen approved angle brief** for this concept (produced first by `/ssc.ads-brief`, approved in the dashboard). **Every read and every generate call keys on it.**

If `idea_id` or `brief_id` is missing, ask the operator for it (one question) before dispatching — do not invent one. There is **no `date` selector**: a `brief_id` is idea-scoped, so the producer always takes an explicit `idea_id`.

Optional (passed through unchanged — this command **never resolves or chooses** a chain or a product itself):

- **Chain** (`chain: <background_creative_id>`) — a brief carries **N parallel visual chains**, each rooted at one **approved `background`** creative. This names the chain to advance. Not needed when the brief has 0 approved backgrounds (the background layer is chain creation) or exactly 1 (unambiguous — the skill resolves it without asking). With **≥2** approved backgrounds and no `chain`, the skill **stops and asks** which one.
- **New chain** (`new-chain`) — mint an **ADDITIONAL** chain: generate 3 fresh background candidates **even though the brief already has an approved background**. This is the **only** way the background layer runs once a root exists — so a plain re-run never burns credits on new roots. `chain` + `new-chain` together is contradictory and stops.
- **Product** (`product: <creative_id>`) — which **approved brief-level product** the composite should use. Only needed when the brief has **more than one** approved product (with several and no selector, the skill **stops and asks** which; with exactly one, it uses it without asking).
- **Revision note** (`revise: <note>`) — a free-text correction for the **active layer of the selected chain**. It makes the skill **rewrite** that layer's prompt and generate ONE fresh candidate. Without it, a layer with pending drafts in that chain simply stops.
- **Model** (`model`) — a fal model id. Omit it and the server default governs.
- **Composite hints** — `layout_hint` (product-placement direction) and `hard_paste` (paste the real packaging's pixels unaltered). **Composite layer only.**
- **Period** (`period`, `YYYY-MM`) — informational only; the month the concept belongs to, used when pointing the operator at `/ad/[month]/[id]`.

**Removed inputs** — if the operator supplies them, say so and drop them: `uploaded_media_id` / `uploaded_media_ref` no longer exist. `generate_model` is **generated-only**; placing a real model photograph in the scene is a **dashboard** action. There is likewise **no `image_content_id`** and **no `n`**.

This command is the **visual-production half** of the Ads pipeline. Its **three preconditions**, checked in order, are: an **approved ad concept**, an **approved angle brief** for it, and **≥1 approved `copy`** for that brief (`/ssc.ads-produce <idea_id> <brief_id> copy` produces the copy; the operator approves it). **`image_content` is NOT a precondition and is never read** — it is the **on-image overlay text** the dashboard applies **over the finished visual at a later stage**, produced separately by `/ssc.ads-produce <idea_id> <brief_id> image_content`. It has nothing to do with this command. The producer operates **per concept**, **per angle (brief)**, **per chain**, and **per layer**, never on a whole plan: it reads **no** `channel_plan` gate flags, and there is no `/ssc.ads-plan` precondition beyond those three. **Phase 1 wires only the ad channel** — a non-ad idea stops cleanly (post/youtube visual flows are a later phase).

## What to do

This command is a thin entry point — it holds **no** orchestration logic. It dispatches the single visual producer **`ssc-image`** (`idea_id`, `brief_id`, optional `chain` / `new-chain` / `product` / `revise` / `model` / `layout_hint` / `hard_paste` / `period` passthrough) for the resolved concept, then stops. **It does not resolve chains, does not pick one, and does not pick a product** — that is the skill's job, and the skill stops and asks rather than guessing.

`ssc-image` is a **state-driven, per-layer stepper with CHAIN-SCOPED state**: on each invocation it reads that **brief's** creatives, groups them into **chains by the lineage every row already carries** (an **approved `background`** is a chain **ROOT** and its id IS the chain id; a `model` belongs to the chain of its `background_creative_id`; a `composite` to the chain of its `scene_creative_id`; a `product` is **brief-level**, shared by every chain), **selects ONE chain**, works the **single next open layer of that chain** — **`background` → `model` → `product` → `composite`** — saves DRAFT creative(s), and **stops**.

| The skill does | Then the operator… |
|---|---|
| Checks the three preconditions in order — approved ad concept → `brief_id` is an **approved** angle brief of it → **≥1 approved `copy`** for that brief (stopping with the exact unmet precondition and next action otherwise; it never reads `image_content`). Resolves the brief's creatives into **chains**, **selects ONE** (stopping to **ask** when there are ≥2 chains and no `chain` argument), and works the first layer **of that chain** without an approved creative — **authoring the full scene prompt itself** (it reaches the image engine verbatim), grounded in the brief's angle + the approved copy (a **meaning** source — its words are never named in a prompt) + the persona detail doc + the brand/visual + compliance KB: **background** (**chain creation** — 3 calls, 3 **distinct readings** of the angle; approving one **mints a chain root**), **model** (exactly ONE candidate generated **INTO the selected chain's ROOT** background), **product** (**brief-level, upload-only** — stops and asks the operator to upload + approve the real packaging photo; stops and asks **which** when several are approved), **composite** (exactly ONE, placing the approved product into **this chain's** approved scene — **never across chains** — with optional `layout_hint` / `hard_paste` as engine-request fields). Saves DRAFT creatives and **stops**, naming **which chain it advanced** and how many chains the brief now has. Each layer runs only when every earlier layer **of that same chain** has an approved creative. | Opens `/ad/[month]/[id]`, **reviews / approves** (or **discards**) the saved DRAFT creative(s) for that layer, then **re-runs `/ssc.image <idea_id> <brief_id> chain: <root id>`** — the skill detects the newly-approved layer of that chain and works the next one. If a pending draft misses the mark, re-runs with **`revise: <note>`** instead. To start another parallel chain, re-runs with **`new-chain`**. |

**A brief holds N parallel chains, and the skill never guesses which one.** With **≥2** approved backgrounds and no `chain`, it **stops and lists every chain** (root background creative id, a one-line gist of that background's prompt, and how far the chain has got) and asks which to advance — it never picks by recency or any other heuristic, and it generates nothing on that run. An invalid `chain` (or an invalid `product`) stops the same way, naming the valid ones. Each generate call spends real credits, so an extra question is always cheaper than a wrong image.

**Don't like a pending draft? Re-run with `revise: <note>`.** The skill then **rewrites that layer's prompt — of the selected chain** — applying the note and issues ONE fresh generate call; never a blind re-roll, never a second batch.

**This flow bakes no text into any visual** — every prompt keeps a clean, evenly-toned reserved zone by describing it **positively** (per the standing composition rule in the visual KB), never by asking for text's absence, and never by naming the ad copy. The producer works **one layer of one chain per run**, reads **no** channel_plan gates, and **saves DRAFT creatives immediately** (no in-chat candidate presentation, no in-chat revise loop). If the active layer of the selected chain already has an unapproved DRAFT and no `revise` note was given, the skill stops and asks the operator to approve/discard it first — no second batch, no second candidate in flight **in that chain** (another chain's pending draft neither blocks nor advances it).

**A CHAIN completes; the BRIEF never does.** A chain is complete when it has an **approved composite**. The skill then stops, names **that chain**, reports how many chains the brief now has, and offers the two real next actions — start another chain with **`new-chain`**, or pick the **hero image** in the dashboard. It never reports the *brief* complete, and it never sets the cover.

## Governance

Nothing auto-approves, auto-applies, auto-discards, sets a cover, or auto-publishes. `ssc-image` **saves DRAFT `creative` rows** and stops; the operator reviews / approves / discards each layer on the `/ad/[month]/[id]` page. **"Save" persists drafts; it is NOT approval** — it never flips a gate, and on the `background` layer **approving is also what curates a candidate into a chain ROOT**, so it is doubly the operator's. Propose-only (hard rule): the producer never calls `approve` (the ONLY gated promotion, denied to agents by the approval hook), never uses `edit` to demote / unapprove / **discard** a creative (discarding is the operator's call), **no `set_cover`** (which approved composite is the hero image is the operator's dashboard choice — it is not in the skill's tools), no `reorder_gallery`, no publish, no `update_budget` — it never uploads the product, never writes a prompt row directly, and never bakes ad text into a visual. It **never guesses a chain or a product**: ambiguity is a STOP-and-ask, never a silent pick, and it **never composes one chain's scene with another chain's model**. The skill talks **only** to BrandOS MCP tools on the `ssc` surface — **never** a third-party image API, not even when a BrandOS call fails. **Each generate call spends fal credits**, so every unmet precondition, ambiguity, or server error is a clean STOP with the exact next action, never a retry-around or a silently-degraded run. All operator-facing prose and persisted notes are **Vietnamese** (image-model prompts are free-form — usually English renders best). Producing requires `edit` (plus `view` for the resolve/precondition reads); approving a draft later requires `approve` on the page.

## After it runs

After the skill saves a layer's DRAFT creative(s) — its summary **names the chain it advanced** (the root background creative id) and how many chains the brief now has — point the operator to `/ad/[month]/[id]` for the concept to **review / approve** the active layer (`background` → `model` → `product` → `composite`) — or **discard** it, or re-run with `revise: <note>` for a re-authored prompt — then **re-run this command** for the same `idea_id` + `brief_id`, passing **`chain: <root id>`** when the brief has more than one chain, to work that chain's next layer. Approving a **background** is what **mints a chain root**. The `product` layer is the operator's own: they upload the **real** packaging photo and approve it in the dashboard (it is **brief-level** — one approved product serves every chain), then re-run.

When a chain has an **approved composite** it is **complete** — but the **brief is never "complete"**: start another parallel chain any time with **`/ssc.image <idea_id> <brief_id> new-chain`** (3 fresh background candidates → approve one → a new root), and pick the **hero image** among the approved composites in the dashboard (`set_cover` is yours, not the skill's). To work a **different angle**, approve a different brief (`/ssc.ads-brief <idea_id>`), get an approved `copy` for it via `/ssc.ads-produce <idea_id> <brief_id> copy`, and pass that `brief_id`.
